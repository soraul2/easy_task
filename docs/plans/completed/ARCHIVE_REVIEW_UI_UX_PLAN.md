# 기록·회고 UI/UX 개선 계획

작성일: 2026-07-24
대상: iPhone `기록` 탭과 `회고 작성/수정` 시트

## 1. 목표와 변경 경계

기록을 찾고 하루를 회고하는 과정에서 사용자가 다음 행동을 바로 이해하고, 빈 입력 화면의 부담 없이 짧게라도 기록을 시작할 수 있게 한다.

- SwiftData 스키마, CloudKit, 백업 형식, 회고 저장 규칙은 변경하지 않는다.
- 기존 `ArchiveQuerySession`의 bounded query와 페이지네이션을 유지한다.
- macOS 기록·회고 화면은 동작과 데이터 의미를 그대로 유지한다.
- 앱의 기존 테마 토큰과 SF Symbols를 사용한다.

## 2. 근거가 된 원칙

### Apple Human Interface Guidelines

- [Human Interface Guidelines](https://developer.apple.com/design/human-interface-guidelines): 콘텐츠와 컨트롤의 명확한 위계, 플랫폼 일관성
- [Layout](https://developer.apple.com/design/human-interface-guidelines/layout): 읽기 순서에 따른 중요 정보 배치, 정렬, 점진적 공개, Dynamic Type 대응
- [Accessibility](https://developer.apple.com/design/human-interface-guidelines/accessibility): iOS 기본 44×44pt 터치 영역, 제스처 대안, 명확한 접근성 레이블
- [Writing](https://developer.apple.com/design/human-interface-guidelines/writing): 짧고 행동 중심인 문구, 화면 목적에 맞는 정보 우선순위
- [Search fields](https://developer.apple.com/design/human-interface-guidelines/search-fields): 콘텐츠 탐색 맥락에 맞는 시스템 검색 사용

### Toss 디자인·라이팅 원칙

- [토스 디자이너가 제품에만 집중할 수 있는 방법](https://toss.tech/article/toss-design-system): 일관된 컴포넌트, 큰 글자와 VoiceOver를 포함한 접근성 내재화
- [토스의 8가지 라이팅 원칙들](https://toss.tech/article/8-writing-principles-of-toss): 명확하고 간결하며 친근한 표현, 사용자를 존중하는 정보 전달
- [좋은 에러 메시지를 만드는 6가지 원칙](https://toss.tech/article/21021): 현재 상황과 다음 행동을 함께 안내하는 피드백
- [시각장애인에게 ‘읽는 순서’가 왜 중요할까?](https://toss.tech/article/voiceover_usability): 레이블·역할·상태와 자연스러운 읽기 순서

PlanBase에는 Toss의 시각 스타일을 그대로 복제하지 않고, 위 원칙 중 정보 위계·간결한 문구·한 번에 하나의 핵심 행동·접근성을 Apple 시스템 UI에 맞게 적용한다.

## 3. 현재 문제

1. 기록 목록이 카드부터 바로 시작해, 사용자가 이 화면에서 무엇을 얻을 수 있는지 맥락이 약하다.
2. 각 기록의 날짜·회고·완료 작업 정보가 같은 위계에 놓여 빠르게 훑기 어렵다.
3. 날짜 보드 이동이 아이콘만으로 표현되어 기능을 추측해야 한다.
4. 회고 작성 시 작업 요약이 입력 영역보다 먼저 크게 펼쳐져 작성 시작을 방해한다.
5. 빈 본문에 자유 기록만 제시되어 무엇을 쓸지 막막할 수 있다.
6. 작업 요약의 가로 카운트와 일부 컨트롤은 큰 글자에서 읽기 흐름이 불안정하다.
7. 이미지 추가와 저장 가능 조건에 대한 안내가 분산되어 있다.

## 4. 개선 설계

### 기록 탐색

- 목록 상단에 현재 불러온 날짜·회고·완료 작업 수를 요약해 화면 목적을 먼저 전달한다.
- 기록 카드의 읽기 순서를 `날짜 → 제목 → 요약 → 회고 본문/사진 → 완료 작업 → 그날 보기`로 정리한다.
- 아이콘 전용 보드 이동을 `그날 보기` 텍스트 버튼으로 바꾸고 기존 접근성 레이블을 유지한다.
- 완료 작업은 기본 접힘 상태를 유지하고 검색 일치 시에만 자동으로 펼친다.
- 필터와 검색은 기존 시스템 패턴과 bounded query를 그대로 사용한다.

### 회고 작성

- 작성 흐름을 `날짜/제목 → 회고 본문 → 작업 요약 → 사진` 순서로 바꿔 핵심 행동인 글쓰기를 먼저 둔다.
- `기억에 남는 일`, `잘한 점`, `배운 점`, `내일의 한 걸음` 질문을 선택형 시작점으로 제공한다.
- 질문을 누르면 중복 없이 본문에 소제목을 추가하고 즉시 본문 입력으로 초점을 이동한다.
- 작업 요약은 기본 접힘으로 바꾸되 상태별 수치는 항상 확인할 수 있게 한다.
- 새 작성과 기존 회고 수정을 내비게이션 제목에서 구분한다.
- 사진 영역은 현재 개수와 최대 개수, 추가 가능 여부를 한 섹션에서 설명한다.

### 접근성·피드백

- 모든 주요 버튼의 최소 높이를 44pt로 유지한다.
- 가로 배치는 `ViewThatFits`로 세로 대안을 제공해 Dynamic Type에 대응한다.
- 장식 아이콘은 VoiceOver 읽기에서 제외하고, 버튼은 레이블·상태·힌트를 제공한다.
- 비어 있는 상태와 실패 상태는 사용자가 다음에 할 수 있는 행동을 함께 제시한다.
- 시스템 검색, 내비게이션, 시트, 사진 선택기를 유지해 플랫폼 관습을 따른다.

## 6. 검증 결과

- `swift test`: 전체 232개 테스트 통과
- `PlanBase-iOS` Debug generic simulator build: 성공
- iPhone 17 Pro에서 기록 이동·회고 진입·취소 보호·저장 관련 UI 테스트: 3개 통과
- 작은 화면의 iPhone 17e에서 기록 이동·회고 진입 관련 UI 테스트: 2개 통과
- 기본 글자 크기와 접근성 초대형 글자 크기의 기록 화면, 기본 글자 크기의 회고 작성 화면을 시뮬레이터에서 확인
- 회고 시트는 모든 섹션을 접근성 트리에 안정적으로 노출하도록 일반 `VStack`을 사용하며, 주요 컨트롤은 44pt 이상 터치 영역과 명시적 레이블을 유지
- SwiftData 스키마, CloudKit, 백업 형식, bounded query, 저장 서비스는 변경하지 않음

## 7. 완료 기준

- 기록 화면에서 날짜별 회고와 완료 작업의 관계를 첫 화면에서 이해할 수 있다.
- 보드 이동, 필터 제거, 작업 펼치기, 사진 추가, 저장의 행동명이 명확하다.
- 회고 시트를 열면 작업 목록보다 본문 입력이 먼저 보이고, 질문 한 번으로 작성이 시작된다.
- 기존 데이터 저장·검색·페이지네이션·이미지 처리 결과가 바뀌지 않는다.
- 관련 단위 테스트, iOS 빌드, UI 테스트가 통과한다.
- 기본 및 접근성 글자 크기에서 핵심 정보와 컨트롤이 잘리거나 겹치지 않는다.
